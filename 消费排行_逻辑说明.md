# 消费排行 · 逻辑说明（评审版 v1）

> 模块：仪表盘 › 消耗分析 › 消费排行（总览页同步复用）
> 一句话目标：让租户管理员快速看出"钱花在哪、谁是大户"，并顺着组织层级逐层下钻定位到责任实体。

---

## 1. 数据层级（关键前提）

严格三级层级，逐级**一对多**，无歧义：

```
部门  ⊃  用户  ⊃  API Key
（1 个 Key 只归属 1 个用户；1 个用户只归属 1 个部门）
```

- 本模块只统计"租户内部可治理实体"（部门 / 用户 / API Key）。
- **不含**请求级"终端用户"（应用上报的 user 参数，与 Key 为多对多）——如需按客户分摊，另立独立视图，不混入本排行。
- 每条消耗记录字段：`部门, 用户, API Key, model, provider, tokens, 费用`。

---

## 2. 三个排行维度

| 维度 | 一行代表 | 口径 | 行内副信息 |
|---|---|---|---|
| 部门 | 一个部门 | 聚合其名下全部用户与 Key 的消耗 | X 名用户 · Y 个 Key |
| 用户 | 一个用户 | 聚合其名下全部 Key 的消耗 | 所属部门 · Y 个 Key |
| API Key | 一个 Key | Key 级明细（最细粒度） | 所属部门 · 所属用户 |

> 颜色统一按**部门**着色：无论看哪个维度，每条的颜色都对应其部门，一眼能看出归属。

---

## 3. 指标与排序

- 指标可切换：**费用（¥）** / **Token**，默认费用。
- 始终按当前指标**从高到低**排序。
- 数值格式：费用 = ¥ + 千分位 + 2 位小数；Token = 百万（M），2 位小数。
- 占比 % = 该条 / **当前作用域合计**（下钻后即为该父级范围内的合计）。

---

## 4. 交互逻辑

1. **起始层级切换**（部门 / 用户 / API Key）：跳到该粒度的**全局扁平排行**，清空下钻路径。
2. **逐层下钻**：点击任意条目 → 进入其子层级
   - 部门 → 该部门的**用户**排行
   - 用户 → 该用户的 **API Key** 排行
   - API Key 为叶子层，不可再下钻。
3. **面包屑**：`全部 › 研发 › usr_rd_01`，点击任意层级可返回。
4. **Top N + 查看全部（超过 5 项的交互，见下节）**：卡片只展示 **Top 5**，不展示"其他"汇总条。

---

## 5. 超过 Top 5 的交互（部门 / 用户 / API Key 通用）

任一层级条目数 > 5 时（含下钻进某部门后其用户 > 5、某用户 Key > 5 的情况），逻辑一致：

- 卡片固定只展示 **Top 5** 条形（保持视觉聚焦头部），右下角出现 **"查看全部 N 项 →"**。
- 点击后打开**全量弹窗**，包含：
  1. **完整列表**：当前层级全部 N 项，默认按当前指标降序；
  2. **可排序**：点 Token / 费用 表头切换排序；
  3. **可分页**：每页 10 条（底部显示"共 N 项"）；
  4. **搜索框**：按名称（部门名 / 用户 / API Key）实时过滤，长列表快速定位；
  5. **可继续下钻**：非叶子层（部门 / 用户）点击名称即从全量列表下钻到下一层级并关闭弹窗，面包屑同步更新——长列表里也能直接钻下去，不是死胡同。
- ≤ 5 项时：全部条形直接展示，不出现"查看全部"入口。
- 弹窗内的列表同样受全局筛选与下钻父级约束（与卡片同一作用域）。

---

## 6. 与全局筛选的联动（重点）

排行**受页面全局筛选约束**，与看板其余卡片同一作用域，避免"别处过滤了、排行没过滤"的不一致。

- 支持的筛选维度：**model / user / API Key / provider**。
- 匹配规则：**同一维度内 OR，跨维度 AND**（例：model=A 或 B，且 provider=OpenAI）。
- 过滤后卡片显示提示：`已按筛选过滤 · N 条记录`。
- 筛选后无数据：显示空态 `当前筛选条件下暂无消耗数据`。
- 下钻与筛选叠加：先按全局筛选缩小数据集，再按下钻父级进一步限定。

---

## 7. 展示与计算规则（明细）

| 项 | 规则 |
|---|---|
| 名次徽章 | 前 3 名用部门色填充，其余浅灰 |
| 条形 | 宽度 = 该条值 / 当前层级最大值；颜色 = 部门色 |
| 数值列 | 指标值 + 占比%（占比 = 该条 / 当前作用域合计） |
| 下钻提示 | 非叶子层每行末尾显示 `›`，整行可点 |
| 面包屑汇总 | 显示"当前按 X · N 项 · 合计 ¥…" |

---

## 8. 边界与业务规则

- **无 1:1 歧义**：用户↔Key、部门↔用户均为多对一，名称列与归属列不会自相矛盾。
- **筛选收敛为单实体**：若筛选把当前维度收敛到 1 条（如筛了某用户又按"用户"排），按"如实展示 1 行"处理（**待确认**是否自动下钻到其 Key 层）。
- **占比口径**：始终相对"当前作用域合计"，而非全局总额，保证下钻后百分比自洽。

---

## 9. 待确认（评审决策点）

1. **数据来源**：各字段（含 model/provider）的取数接口、刷新频率、时区、聚合粒度——当前为前端 mock，需与后端对齐。
2. **Top N 值**：当前 Top 5，是否可配置 / 改 Top 10。
3. **单实体收敛**：筛选/下钻后只剩 1 条时，是否自动进入下一层级。
4. **权限范围**：哪些角色可见费用/部门级数据（管理员 vs 普通成员）。
5. **导出**：明细弹窗是否需要导出 CSV。
