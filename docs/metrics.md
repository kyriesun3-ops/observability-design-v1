# 租户管理平台 · 可观测分析指标清单

> 适用范围：租户管理平台「可观测 - 分析」全部模块。
> 视角：面向**租户管理员 / 租户**，回答"钱花得值不值、用得好不好、谁在用"。
>
> **铁律（全局筛选约束）**：页面顶部存在公共的「搜索 filter」与「时间筛选下拉」，
> 所有指标的数值均为**当前筛选窗口内的纯聚合值**，不得隐含固定时间基准。
> 不提供固定周期环比（未定义统一统计周期），趋势变化由折线图本身体现。
>
> 例外：标记为「存量快照」的指标为基线参照，不随时间窗收缩。

## 成本口径：时间窗摊销成本（amortized cost）

所有成本类指标（KPI 总成本、成本构成、各模块成本卡）**均跟随时间筛选**，
展示"归属于所选窗口"的成本。两类成本性质处理方式不同：

- **用量型**（API 路由按 token、GPU 容器实例按机时、云电脑按运行时长）：直接累加窗口内实际发生费用。
- **订阅 / 固定型**（云硬盘按容量包月、云电脑包月套餐）：按**日费率摊销**
  （月费 ÷ 当月天数 × 窗口内天数）分摊到窗口。

好处：无论选 24 小时还是 90 天，所有成本数字可加、自洽，占比图始终反映窗口内真实构成。

## 全局搜索筛选维度

参考 Portkey（Model / Cost / Tokens / Cache / Status / Meta / Avg.Weighted Feedback /
Providers / API Key / Trace Id），按本平台多资源特性调整为：

`Resource(资源类型) · Model · Cost · Tokens · Cache · Status · User · API Key · Provider · Trace Id`

> 其中 `Resource` 为本平台新增（云电脑 / GPU 容器实例 / API 路由 / 云硬盘），是 Portkey 所没有的核心维度。

## 1. 缓存命中（Cache）

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| Cache Hits | 窗口内命中缓存的请求数，区分精确命中 / 语义命中 | `count(simple_hit) + count(semantic_hit)` |
| Cache Hit Rate | 缓存命中率 | `命中请求数 / 总请求数 × 100%` |
| Cache Speedup | 缓存带来的响应加速比 | `(1 − 缓存平均延迟 / 未命中平均延迟) × 100%` |
| Cache Savings | 缓存节省的成本 | `Σ(命中请求若实际调用模型的预估成本)` |
| Avg Cache Latency | 命中缓存请求的平均延迟 | `avg(命中请求的响应时间)` |

## 2. 报错分析（Errors）

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| Error Rate | 错误率 | `错误请求数 / 总请求数 × 100%` |
| Error Count | 错误请求数，按 error code 分组堆叠 | `count(status != success) group by error_code` |
| Error Types | 错误类型种数（归类后） | `distinct(error_category)` |
| Rescued Requests | 经重试 / 降级后被挽救成功的请求数 | `count(首次失败 且 最终成功)` |

## 3. 风险拦截（Risk）

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| 拦截数统计 | 触发风险规则被拦截的请求数 | `count(intercepted = true)` |
| 拦截率 | 拦截请求占比 | `拦截请求数 / 总请求数 × 100%` |
| 风险类型分布 | 各风险类型（数据泄露 / 有害内容 / 注入等）拦截数占比 | `count(intercepted) group by risk_type` |
| Top 风险用户 | 拦截次数最高的用户榜 | `count(intercepted) group by user, 降序 top N` |
| 拦截记录 | 拦截明细（时间 / 用户 / 类型 / 片段） | 明细行，按时间倒序 |

## 4. 延迟分析（Latency）

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| 平均延迟（Total Latency） | 请求发起到响应完成的总耗时，支持 P50 / P95 / P99 | `percentile(end_time − start_time, P)` |
| 首字延迟 TTFT | 请求发起到收到首个 token 的耗时 | `avg(first_token_time − start_time)` |
| 延迟类别分布 | 总延迟拆成 网络 / 推理 / 排队 三段 | 各段耗时堆叠 |

## 5. 多媒体模型（Multimodal）

> 计量单位差异：图像=张、音频=分钟、视频=秒。为支持单页跨模态聚合，
> 用量与成本卡统一归一化为「次 / $」，原生单位下钻留待 V2。

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| 多模态调用量 | 图 / 音 / 视频调用次数，按模态堆叠（统一用"次"以便跨模态聚合） | `count(request) group by modality` |
| 多模态成本 | 各模态成本归因（视频单价高，重点观察） | `Σ(cost) group by modality` |
| 异步任务状态 | 任务成功率 + 各状态(成功 / 处理中 / 失败 / 超时)分布 | 成功率 = `成功任务数 / 总任务数 × 100%` |
| 平均处理时长 | 异步任务的处理耗时，支持 P50 / P95 / P99 | `percentile(processing_time, P)` |
| 任务类型分布 | 文生图 / 图生图 / TTS / STT / 文生视频 / 视频理解 占比 | `count(task) group by task_type` |

**同步 / 异步划分**（影响指标口径）：
- **同步·理解类**（文+图、文+视频、音频+文）：输出为 token，复用 LLM 路由分析范式（TTFT、Token、成本）。
- **异步·生成处理类**（纯视频生成、TTS / STT 音频处理）：核心为「异步任务状态」与「处理时长」。

## 6. Desk 云桌面（Desk）

> 计费：按规格 + 时长。模式：持久专属（每台桌面绑定到固定使用者）。
> 租户最关心：成本（hero）、使用体验。
>
> **双开通路径**：桌面可由①租户管理员开通分配，也可②用户在 client 端自购（C 端场景）。
> 因此"已分配"无法作为全量基线——状态类指标一律按**桌面真实系统状态**统计，不依赖开通来源。
>
> **桌面状态枚举**（对接现有系统）：运行中 / 已关机 / 已释放（待补全：是否还有 创建中 / 已分配 / 异常 等）。
> 其中「已关机」= 占用未释放、可能仍在产生存储费，是租户的省钱抓手。

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| Desk 总成本（按规格）★hero | 按计费规格(标准 / GPU / 高内存)归因的总成本，跟随时间筛选（摊销口径） | `Σ(规格日费率 × 窗口内时长) group by spec` |
| 资源利用率 | 开机时间里真正活跃操作的比例（衡量挂机浪费）。不区分是否分配——桌面可能是用户自开 | `活跃操作时长 / 开机时长 × 100%` |
| 桌面状态分布 | 按系统真实状态(运行中 / 已关机)统计桌面数随时间构成，不依赖开通来源 | `count(desk) group by status` |
| 连接质量 RTT | 连接往返延迟 P50 / P95（>200ms 体验明显下降） | `percentile(round_trip_time, P)` |
| 桌面使用明细 | 单桌面 使用者 / 规格 / 时长 / 利用率 / 状态 | 明细行，利用率 < 20% 标红高亮，状态对接系统枚举 |

## 7. 用户使用（Users · 全平台聚合，对齐 Portkey User 指标）

> 范围：全平台聚合，覆盖 云电脑 + GPU 容器实例 + API 路由 + 云硬盘 四类资源。
> 目的：成本可见性（showback）+ 识别重度用户。V1 不做组织下钻，但列表保留「部门」列为 V2 铺路。
>
> 命名：用「用户 / User」而非「员工」——因为存在用户在 client 端自行开通资源的 C 端场景，
> 使用对象不一定是受雇员工，且与「用户中心」术语一致。
>
> Portkey 对标：Unique Users（折线）、Request per User（折线）、用户请求列表（user / requests / cost）。
> 本设计在其基础上把单一 cost 扩展为按四类资源拆分的成本构成。
>
> **活跃用户定义**：活跃用户数 = **所选时间筛选窗口内**（卡片副标题显示 `Data from xxx to xxx`），
> 发生过任意被计量行为（API 请求、桌面会话、容器使用等任一）的**去重用户数**。
> 该值完全跟随时间筛选变化，无固定统计周期；切换时间窗口数值随之重新聚合。

| 指标 | 解释 | 计算方式 |
|------|------|---------|
| 活跃用户数（Unique Users） | 所选时间窗口内有任意被计量行为的去重用户数，副标题显示 `Data from xxx to xxx` | `distinct(user_id where 窗口内有行为)` |
| 全平台总成本 | 四类资源(云电脑 + GPU 容器实例 + API 路由 + 云硬盘)摊销成本合计，跟随时间筛选 | `Σ(amortized_cost) across all resource types` |
| 人均成本 | 平均每个活跃用户的成本 | `全平台总成本 / 活跃用户数` |
| 成本构成（按资源类型） | 四类资源的成本占比（环形图），跟随时间筛选 | `Σ(amortized_cost) group by resource_type`，占比 = 各类 / 总和 |
| 用户使用趋势 | 独立用户数(Unique Users) & Request per User 走势，X 轴随筛选窗口 | 双轴时序；Request per User = `窗口内总请求数 / 活跃用户数` |
| 用户请求列表 | 每用户 请求数 + 各资源花费 + 总成本（含部门列） | `count(request)` 与 `Σ(cost) group by user, resource_type` |

> 注：本模块所有数值均为所选时间窗口内的聚合值，**不提供固定周期环比**
> （未定义统一统计周期，环比基准会与任意时间窗冲突）。趋势变化通过折线图本身体现。

## 待定 / V2 规划

- **桌面状态枚举对齐**：当前用 运行中 / 已关机 / 已释放，需与系统完整枚举对齐（是否含 创建中 / 已分配 / 异常 等）。
- **自购桌面成本归属**：用户 client 端自购桌面的成本是否计入租户账单，待确认——影响成本卡是否纳入自购部分。明细表已加「开通来源」列（租户分配 / 用户自购）作为过渡。
- **摊销成本落地**：订阅 / 固定型资源（云硬盘、包月套餐）的日费率摊销需后端账单支持，前端按摊销值展示。
- **全局筛选器 UI**：完整的多维筛选器（Resource / Model / Cost / Status / User / API Key / Provider / Trace Id）为独立工作量，当前仅占位文案，未实现交互。
- **多模态原生单位下钻**（图按分辨率张数 / 音频按分钟 / 视频按秒）：V2。
- **异步任务监控扩展**（排队时长 / 当前积压 Backlog）：V2。
- **组织下钻**（按部门 / 团队查看更小范围数据分析）：V2，列表已保留部门列。
- **Desk 体验扩展**（登录时长 / 掉线率）：V2。
