# 需求澄清：平均延迟 & 平均生成时长 · 计算公式与口径

> 面向：研发 / 测试 ｜ 涉及卡片：`LatencyCard`（平均延迟）、`GenTimeCard`（平均生成时长）
> 依据：`docs/metrics.md` 第 4、5 节 + 原型截图 ｜ 日期：2026-07-13

## 0. 核心结论（一句话）

**卡片上的每一个数字都是"对原始请求明细直接取分位数"，任何位置都不允许对分位数再做平均。** 头部大数字、总览曲线、按模型曲线，三者只是"取分位数前的过滤范围"不同。

## 1. 基础字段定义

| 字段 | 含义 | 备注 |
|------|------|------|
| `start_time` | 网关收到请求的时刻 | LLM 请求 |
| `end_time` | 响应完成时刻（流式 = 最后一个 token 发出） | LLM 请求 |
| `latency` | `end_time − start_time`（ms） | 请求级明细 |
| `submit_time` | 生成任务提交时刻 | 多模态异步任务 |
| `complete_time` | 任务进入终态（成功）时刻 | 多模态异步任务 |
| `processing_time` | `complete_time − submit_time`（s） | 任务级明细，**含排队时间**（默认口径，见 §4-Q3） |
| `modality` | image / video / audio | 生成时长卡的模态过滤字段 |

## 2. 平均延迟卡（LatencyCard）三层数值

设时间筛选窗口为 `[T0, T1]`，用户在下拉框选择分位 `P ∈ {P50, P95, P99}`。

| 位置 | 公式 | 说明 |
|------|------|------|
| ① 头部大数字（如 `P50: 1.2s`） | `percentile(P, 窗口内全部请求的 latency)` | **全窗口、全模型合并后的明细直接取分位**。不是各日 P50 的均值，不是各模型 P50 的均值 |
| ② 总览模式·曲线每点 | `percentile(P, 当日全部请求的 latency)` | 按天分桶，桶内明细取分位 |
| ③ 按模型模式·曲线每点（tooltip 中 gemini-1.5-pro 1,317 ms 等） | `percentile(P, 当日 && model=m 的请求 latency)` | 先按 天×模型 过滤，再取分位 |

切换 P50/P95/P99：三层数值同时切换，仅改变 `P`，过滤逻辑不变。

## 3. 平均生成时长卡（GenTimeCard）三层数值

在 LatencyCard 基础上多一个**模态过滤**（图像/视频/音频 tab），公式相同，把 `latency` 换成 `processing_time`：

| 位置 | 公式 |
|------|------|
| ① 头部大数字（如 `视频 P50: 5s`） | `percentile(P, 窗口内 modality=选中模态 的全部任务 processing_time)` |
| ② 总览模式·曲线每点 | `percentile(P, 当日 && modality=选中模态 的任务 processing_time)` |
| ③ 按模型模式·曲线每点 | `percentile(P, 当日 && modality=选中模态 && model=m 的任务 processing_time)` |

头部大数字**跟随模态 tab 联动**，前缀显示当前模态名（如"视频 P50: 5s"）。

## 4. 口径决策（默认值 + 待研发确认）

| # | 问题 | 默认口径 | 状态 |
|---|------|---------|------|
| Q1 | 失败请求是否计入延迟？ | **不计入**。延迟只统计成功完成的请求；报错请求的耗时无业务意义且会污染分位数 | 待确认 |
| Q2 | 分位数算法 | 与存储引擎默认一致即可（如 ClickHouse `quantile` 为近似算法），但**前后端/测试必须固定同一种**，QA 校验时用同一算法对拍 | 待确认 |
| Q3 | 异步任务计时起点 | 默认从 `submit_time` 起算（含排队），因为这是用户感知的真实等待。若产品要区分"排队 vs 推理"，V2 再拆段 | 待确认 |
| Q4 | 失败/超时的生成任务 | 不计入 processing_time 分位数（同 Q1） | 待确认 |
| Q5 | 按天分桶的时区 | 租户所在时区（与全站时间筛选器一致），桶边界 `[00:00, 24:00)` | 待确认 |
| Q6 | 单位显示 | latency < 1000ms 显示 `ms`，≥ 1000ms 转 `s` 保留 1 位小数；生成时长统一 `s` | 待确认 |
| Q7 | 空数据 | 某天某模型无请求 → 该点断开（null），不画 0 | 待确认 |

## 5. 禁止的错误实现（研发注意）

1. ❌ 头部大数字 = 曲线各点的平均值（"P50 的平均"没有统计意义）。
2. ❌ 头部大数字 = 各模型 P50 的平均值。
3. ❌ 先按天算 P50 再对 30 天取 P50（分位数不可嵌套/合并，必须回到明细算）。
4. ❌ 用 `avg(latency)` 冒充 P50。

## 6. QA 验收要点

1. **不变量校验（重点）**：全体合并的 P50 必然落在 `[各模型当期 P50 的最小值, 最大值]` 区间内。
   - 截图 1：头部 `1.2s`，各模型 811–1,317 ms → ✅ 合理。
   - 截图 2：头部 `视频 P50: 5s`，但所有视频模型当日 P50 均 ≥ 13.4s → ❌ **数学上不可能**。这是原型 mock 数据的 bug，真实实现中若出现此现象即为计算错误（大概率是 ①头部没跟随模态过滤，或 ②混入了图像任务）。
2. 切换 P50→P95→P99：三层数值单调不减（P95 ≥ P50）。
3. 切换模态 tab：生成时长卡的头部数字、曲线、图例（模型集合）三者同时刷新且模型列表只含该模态的模型。
4. 总览 ↔ 按模型切换：同一天，总览点的值落在各模型点的 min/max 之间（同不变量 1）。
5. 修改时间筛选窗口：头部大数字重新聚合（副标题"数据来自 x 至 x"同步更新），不是缓存的固定周期值。
6. 边界：窗口内 0 条请求 → 头部显示 `—`，不显示 0；单条请求 → P50=P95=P99=该条值。
7. 对拍方法：从明细表导出窗口内原始 latency，用约定的分位数算法（Q2）离线计算，与前端展示值比对（近似算法允许 ±1% 误差，需研发给出误差上界）。
