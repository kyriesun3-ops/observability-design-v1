# 卡片标题 Hover 文案 · 优化对照稿（待你定稿）

> 用法：看「建议」列，直接在上面改；定稿后我回填原型。
> 统一规则：① 时间窗口 → **时间范围**；② 中文统一用全角括号（）；③ 模型相关卡用「涉及模型」，聚合/请求类卡用「统计范围」（此项可由你拍板是否统一成一个词）。

## 一、消耗分析 · 账户额度（全局值，不随时间筛选）

**1. 可用额度**
- 现状：可用额度 = 累计充值 + 赠金 − 累计消费，为账户实时余额，不随时间筛选变化。
- 建议：当前可继续消费的余额 = 累计充值 + 赠金 − 累计消费。为账户实时余额，不随时间筛选变化。

**2. 累计充值**
- 现状：账户开通以来通过付费充值累计到账的金额，不含赠金；为全局累计值。
- 建议：账户开通至今的付费充值到账总额（不含赠金）。为全局累计值，不随时间筛选变化。

**3. 累计消费**
- 现状：账户开通以来在全部模型与服务上累计扣费的金额；为全局累计值。
- 建议：账户开通至今在全部模型与服务上的累计扣费总额。为全局累计值，不随时间筛选变化。

**4. 赠金**
- 现状：平台赠送的代金额度，优先于充值余额抵扣消费；为全局余额。
- 建议：平台赠送的代金余额，消费时优先于充值余额抵扣。为全局余额，不随时间筛选变化。

## 二、消耗分析 · 消耗概览（随时间筛选）

**5. 请求数**
- 现状：所选时间窗口内的总请求数。｜涉及模型：文本 + 多模态全部模型
- 建议：所选时间范围内的 API 请求总数。｜涉及模型：文本与多模态全部模型。
**6. 总 token**
- 现状：所选时间窗口内的 Token 消耗总量，拆分为输入、缓存、输出三部分。｜涉及模型：文本 + 多模态全部模型
- 建议：所选时间范围内的 Token 消耗总量，按输入、缓存、输出三部分拆分。｜涉及模型：文本与多模态全部模型。

**7. 图片生成**
- 现状：所选时间窗口内的图片生成任务数，拆分为成功与失败。｜涉及模型：图像生成模型(文生图/图生图)
- 建议：所选时间范围内的图片生成任务数，按成功与失败拆分。｜涉及模型：图像生成模型（文生图 / 图生图）。

**8. 视频生成**
- 现状：所选时间窗口内的视频生成任务数，拆分为成功与失败。｜涉及模型：视频生成模型(文生视频/视频理解)
- 建议：所选时间范围内的视频生成任务数，按成功与失败拆分。｜涉及模型：视频生成模型（文生视频 / 视频理解）。

## 三、消耗分析 · 趋势与分布

**9. 用量汇总**
- 现状：按时间展示 Token 用量，堆叠呈现输入、缓存、输出三类 Token 的消耗量。｜涉及模型：文本 + 多模态(图像/音频/视频)全部模型
- 建议：所选时间范围内 Token 用量的每日走势，按输入、缓存、输出三类堆叠展示。｜涉及模型：文本与多模态（图像/音频/视频）全部模型。

**10. 消费汇总**
- 现状：按时间展示账户消费金额走势，反映各日实际扣费情况。｜涉及模型：文本 + 多模态(图像/音频/视频)全部模型
- 建议：所选时间范围内消费金额的每日走势，反映各日实际扣费。｜涉及模型：文本与多模态（图像/音频/视频）全部模型。

**11. 按服务商分布**
- 现状：所选时间窗口内消耗按上游服务商归因，可在费用与 Token 维度间切换。｜涉及模型：文本 + 多模态全部模型，按服务商聚合
- 建议：所选时间范围内的消耗按上游服务商归因，可在费用与 Token 两种口径间切换。｜统计范围：文本与多模态全部模型，按服务商聚合。

**12. 按模型分布**
- 现状：所选时间窗口内消耗按具体模型归因，可在费用与 Token 维度间切换。｜涉及模型：文本 + 多模态全部模型，按模型聚合
- 建议：所选时间范围内的消耗按具体模型归因，可在费用与 Token 两种口径间切换。｜统计范围：文本与多模态全部模型，按模型聚合。

**13. 消耗排行**
- 现状：层级为 部门 ⊃ 用户 ⊃ API Key（均多对一）。点击条目可逐层下钻 (部门→用户→Key)，或用右侧切换直接跳到某一粒度的全局排行。按消耗从高到低。｜涉及模型：文本 + 多模态全部模型
- 建议：按部门 / 用户 / API Key 三个层级看消耗排行（层级为 部门 ⊃ 用户 ⊃ API Key，逐级一对多）。点击任意条目可下钻到下一层级，也可用右上切换直接查看某一层级的全量排行；默认按消耗从高到低。｜统计范围：文本与多模态全部模型。

## 四、缓存命中

**14. 缓存命中token数**
- 现状：缓存命中节省的 Token 总量与命中率；曲线展示缓存命中token数走势。命中越多、命中率越高，越能降低重复推理成本。｜涉及模型：文本模型(命中缓存复用的输入/输出 Token)
- 建议：因命中缓存而复用的 Token 总量与命中率；曲线为其每日走势。命中越多、命中率越高，越能省下重复推理的成本。｜涉及模型：文本模型（命中缓存复用的输入 / 输出 Token）。

**15. 缓存节省成本**
- 现状：缓存命中的请求免去了真实模型调用，按其原应产生的费用估算累计节省金额。｜涉及模型：文本模型(命中缓存复用的请求)
- 建议：命中缓存的请求免去了真实模型调用，按其原本应产生的费用估算出的累计节省金额。｜涉及模型：文本模型（命中缓存复用的请求）。

## 五、报错分析

**16. 报错率**
- 现状：报错请求数占总请求数的比例，反映服务整体稳定性。｜涉及模型：文本 + 多模态全部模型的 API 请求
- 建议：报错请求数 ÷ 总请求数，反映服务整体稳定性。｜统计范围：文本与多模态全部模型的 API 请求。

**17. 报错数量**
- 现状：按 HTTP 状态码 (429 限流 / 500 服务端 / 401 鉴权) 堆叠展示的报错次数。｜涉及模型：文本 + 多模态全部模型的 API 请求
- 建议：报错次数，按 HTTP 状态码（429 限流 / 500 服务端 / 401 鉴权）堆叠展示。｜统计范围：文本与多模态全部模型的 API 请求。

**18. 报错类型分布**
- 现状：按错误类型对报错请求归类，定位主要故障来源。｜涉及模型：文本 + 多模态全部模型的 API 请求
- 建议：将报错请求按错误类型归类，帮助快速定位主要故障来源。｜统计范围：文本与多模态全部模型的 API 请求。

**19. 挽救请求数**
- 现状：通过自动重试 / 故障转移成功挽回的报错请求数，反映容错能力。｜涉及模型：文本 + 多模态全部模型的 API 请求
- 建议：通过自动重试 / 故障转移成功挽回的报错请求数，反映系统容错能力。｜统计范围：文本与多模态全部模型的 API 请求。

## 六、延迟分析

**20. 平均延迟**
- 现状：请求端到端总耗时的分位数 (P50/P95/P99)，可切换分位观察长尾延迟。｜涉及模型：文本模型(对话/补全请求)
- 建议：请求端到端总耗时的分位数（P50 / P95 / P99），可切换分位观察长尾延迟。｜涉及模型：文本模型（对话 / 补全请求）。

**21. 平均首字延迟（TTFT）**
- 现状：从发起请求到返回第一个 Token 的耗时，衡量流式响应的初始体验。｜涉及模型：文本模型(流式对话/补全请求)
- 建议：从发起请求到返回第一个 Token 的耗时（TTFT），衡量流式响应的初始体验。｜涉及模型：文本模型（流式对话 / 补全请求）。

## 七、多媒体模型

**22. 多模态调用量**
- 现状：按模态 (图像/音频/视频) 堆叠统计的多模态请求次数，跨模态归一化为「次」。｜涉及模型：多模态模型：图像(文生图/图生图)、音频(TTS/STT)、视频(文生视频/视频理解)
- 建议：多模态请求次数，按模态（图像/音频/视频）堆叠统计，跨模态统一以「次」计。｜涉及模型：多模态模型——图像（文生图/图生图）、音频（TTS/STT）、视频（文生视频/视频理解）。

**23. 生成类型分布**
- 现状：按生成媒体类型 (图片/视频/音频) 统计的产物数量占比。｜涉及模型：多模态生成模型:图像、视频、音频
- 建议：产物数量按生成媒体类型（图片/视频/音频）的占比分布。｜涉及模型：多模态生成模型——图像、视频、音频。

**24. 多模态成本**
- 现状：按模态 (图像/音频/视频) 归因的消费金额，视频单价高需重点观察。｜涉及模型：多模态模型:图像、音频、视频
- 建议：消费金额按模态（图像/音频/视频）归因；视频单价较高，建议重点关注。｜涉及模型：多模态模型——图像、音频、视频。

**25. 平均生成时长**
- 现状：多模态生成任务从提交到产物完成的处理时长分位数(P50/P95/P99)。按模态(图像/音频/视频)切换，默认视频；可在总览与按模型之间切换。｜涉及模型：多模态生成模型：图像、音频、视频
- 建议：多模态生成任务从提交到产物完成的处理时长分位数（P50 / P95 / P99）。可按模态（图像/音频/视频）切换，默认视频；并可在总览与按模型之间切换。｜涉及模型：多模态生成模型——图像、音频、视频。

### 待你拍板的统一项
1. 「涉及模型」与「统计范围」是否统一成一个词（聚合/请求类卡用「统计范围」更准，但与模型类卡不一致）。

3. 语气：是否要更口语/更友好（例如把"归因"换成"按…拆分归类"）。
